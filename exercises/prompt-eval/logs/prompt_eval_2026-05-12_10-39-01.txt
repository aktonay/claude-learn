  Two prompt versions tested on the same dataset:

  PROMPT V1 (naive):
    "Please solve the following task: {task}"
    No formatting instructions. No constraints.

  PROMPT V2 (improved):
    Explicit format rules + assistant prefilling + stop sequence.
    "Respond only with Python, JSON, or Regex. No comments. No markdown."
    Prefills assistant with backtick to steer code output.
    Uses stop sequence to cut off markdown wrapping.

  GRADING (each output gets TWO scores averaged together):

  CODE GRADER (syntax validation):
    JSON  -> json.loads() — returns 10 (valid) or 0 (invalid)
    Python -> ast.parse()  — returns 10 (valid) or 0 (invalid)
    Regex -> re.compile()  — returns 10 (valid) or 0 (invalid)
    Fast, free, deterministic. Checks STRUCTURE only.

  MODEL GRADER (quality evaluation):
    Sends output to API with eval prompt asking for:
    - strengths (1-3 items)
    - weaknesses (1-3 items)
    - reasoning (why this score)
    - score (1-10)
    Uses solution_criteria from test case for targeted evaluation.
    Flexible but costs an API call. Checks QUALITY.

  COMBINED SCORE = (syntax_score + model_score) / 2

======================================================================
  KEY TAKEAWAYS
======================================================================

  1. PROMPT EVALUATION = OBJECTIVE MEASUREMENT
     Instead of 'this prompt feels better', you get numbers.
     V1 scored X/10, V2 scored Y/10 — clear, comparable, trackable.

  2. TWO GRADER TYPES COVER DIFFERENT ASPECTS
     Code grader: 'Is the syntax valid?' (binary, deterministic)
     Model grader: 'Is the solution correct and helpful?' ( nuanced)
     Neither alone gives the full picture. Combined = comprehensive.

  3. ITERATION IS THE POINT
     V1 is your baseline. V2 is your first improvement.
     The delta between them proves whether your changes helped.
     Keep iterating: V3, V4, V5... until scores plateau.

  4. DATASET QUALITY MATTERS
     3 test cases is minimal. Production evals use 50-200+ cases.
     Include edge cases, ambiguous inputs, and realistic scenarios.
     The dataset IS your test suite for prompts.

  5. THE THREE OPTIONS AFTER WRITING A PROMPT:
     Option 1: Test once, ship it.    (risky — breaks in production)
     Option 2: Test a few times, tweak. (better — still misses edge cases)
     Option 3: Run through eval pipeline. (best — objective metrics)

  This exercise implements Option 3.
